## Storage Systems

In this design, multiple storage systems are used to support different business and AI goals efficiently. The transactional data from EHR/EMR databases and billing systems is stored in relational OLTP databases because they provide ACID properties, which are required for healthcare data accuracy and consistency. ICU monitoring devices generate continuous real-time data, so a streaming platform such as Kafka is used for ingestion before storing the data in the data lake. For long-term storage and analytics, a data lakehouse architecture with Bronze, Silver, and Gold layers is used. The Bronze layer stores raw data, the Silver layer stores cleaned and anonymized data, and the Gold layer stores aggregated tables for reporting. This layered approach ensures data quality and supports multiple use cases. Machine learning models for readmission risk use the Gold layer because it contains structured and reliable data. NLP-based patient history search uses the Silver layer because it still contains detailed text data. BI dashboards use the Gold layer for fast query performance, while real-time ICU alerts use the streaming data pipeline to process incoming data quickly.

## OLTP vs OLAP Boundary

In this system, the OLTP boundary ends at the operational databases such as EHR/EMR and billing systems. These systems handle day-to-day transactions like patient records, billing entries, and device readings. They are optimized for fast inserts and updates and must maintain strict consistency. Once the data is extracted using batch ETL or streaming pipelines, it moves into the data lakehouse, which marks the beginning of the OLAP system. The lakehouse is designed for analytics, reporting, and machine learning rather than transactions. The Bronze layer stores raw data, the Silver layer contains cleaned and transformed data, and the Gold layer contains aggregated data used for business intelligence and AI models. Separating OLTP and OLAP ensures that heavy analytical queries do not slow down the operational healthcare systems.

## Trade-offs

One major trade-off in this design is between real-time processing and system complexity. Supporting both batch ETL and streaming pipelines increases flexibility but also makes the architecture more difficult to manage. Real-time ICU alerts require low latency, which means streaming tools like Kafka must be used, while reporting and machine learning require batch processing and cleaned data in the lakehouse. Maintaining both pipelines increases operational cost and requires more monitoring. To mitigate this, the system can use a unified lakehouse architecture where both batch and streaming data are stored in the same storage system. Using standardized data formats and automated ETL workflows reduces complexity. Proper monitoring tools and data validation checks can also ensure that both real-time and batch processes work reliably without affecting system performance.
